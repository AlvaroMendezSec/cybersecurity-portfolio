# Enterprise Security Monitoring Lab: Wazuh SIEM + Active Directory

##  Project Description
This hands-on lab simulates a corporate environment designed to centralize, analyze, and detect security events in real time. The infrastructure integrates a Windows Server-based Active Directory (AD) environment with the open-source Wazuh SIEM, enabling the development of skills in alert triage, advanced endpoint monitoring, and Threat Hunting.

The primary objective of this project is to gain practical, real-world experience in the daily tasks of a SOC (Security Operations Center) Analyst, covering everything from telemetry engineering to the detection of tactics and techniques outlined in the MITRE ATT&CK framework.

## Lab Architecture and Components

- Component / Role	| Operating System      |	Technical Function
- Wazuh Manager     |	Ubuntu Server 22.04   | LTS	Central server. Collects, parses, and correlates logs received through security rules.
- Domain Controller |	Windows Server 2025   |	Domain Controller (KDC Kerberos / Active Directory). Manages identities.
- Victim Endpoint	  | Windows 11 Enterprise	|Domain-joined workstation under continuous monitoring by the Wazuh agent.

## Telemetry Engineering (Auditing Hardening)
To eliminate "blind spots" in the network, advanced auditing policies were configured on the Windows operating systems, ensuring complete visibility into critical events:

Identity Monitoring (Kerberos & NTLM): Activation of advanced policies via secpol.msc to capture credential validation events (Event ID 4776), successful logins (Event ID 4624), and failed logins (Event ID 4625).

Advanced Process Visibility: Activation of detailed process tracking (Event ID 4688) and registry configuration to include the full Command Line (commandLine), allowing auditing of exact arguments executed in consoles (CMD/PowerShell).

## Use Cases and Attack Simulations (MITRE ATT&CK)
Case Study: [Unauthorized Port Scanning]

### Summary

Initial tactical reconnaissance activity (active port scanning) was detected and logged against the infrastructure's Domain Controller (WIN-L1E9NH3SQQA). The attack was executed automatically via bursts of TCP connections within an extremely short time frame (milliseconds), aiming to identify exposed services and map the internal attack surface.

### Objectives

Generate controlled suspicious activity.

Analyze the telemetry collected by Wazuh.

Identify relevant events.

Document the investigation process.

### Evidence

The compelling forensic evidence is centralized within the Wazuh SIEM through the mass indexing of Event ID 5152 (Filtering Platform Packet Drop), correlated with the attacker's source IP address. The local Windows Server Event Viewer corroborated the anomaly prior to its ingestion into the SIEM.

### MITRE ATT&CK

**TTP: T1595 – Active Scanning**

### What Happened?

A network reconnaissance attempt (Network Scanning) was detected against the infrastructure's Domain Controller. An unauthorized host on the network segment (192.168.139.132) performed a TCP port scan to identify exposed services and potential entry vectors.

### Evidence Demonstrating the Activity
The primary evidence is the burst of Event ID 5152 events generated by the Windows Filtering Platform (WFP). The logs show multiple connection attempts toward sensitive ports (such as port 53 - DNS) and random ports within a millisecond time frame, all originating from the same external IP address.

### Information Provided by the Logs
The Wazuh logs (fed by the Windows Security channel) provide the following details:

- Attack Source: IP 192.168.139.132 (Kali Linux)

- Victim: Host WIN-L1E9NH3SQQA (Windows Server)

- Technical Vector: Use of Stealth-type packets (likely an Nmap SYN scan)

- Firewall Action: Automatic packet blocking and dropping (Packet Drop)

### Information Obtained by the Attacker
Because the Windows Firewall was set to "Drop" mode, the attacker received "Filtered" responses. This means the attacker likely could not confirm which services were open, but they did confirm that the host was online and protected by an active firewall.

### Lessons Learned
During the investigation, it was discovered that Event ID 5152 was being filtered by the Wazuh agent configuration (with the goal of reducing log volume), which prevented its visibility within the SIEM. It was necessary to review the ossec.conf file to analyze the observed behavior.
